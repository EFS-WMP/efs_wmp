# Phase 0

## Purpose & Scope
This Phase 0 document establishes the system boundaries and ownership for the ITAD operations system, ensuring unambiguous separation of responsibilities between Odoo, ITAD Core, and Routific to prevent data conflicts and integration issues.

**SoR Lock:** Odoo is the system of record for scheduling, day routes, and dispatch execution; ITAD Core is the system of record for compliance, receiving, processing, custody, evidence, reconciliation, lots, shipments, disposition, certificates, and settlement; Routific remains optimizer-only. Acceptance/dispatch execution commits to Odoo. ITAD Core receives compliance artifacts later via pickup_manifest → BOL → receiving. Routific proposals are stored/versioned in Odoo; ITAD Core may only store routific_job_id/input_hash for traceability.

## Assumptions / Non-Goals
Assumptions include API-based integration between systems and adherence to single-writer principles. Non-goals include detailed API specifications, implementation code, or performance benchmarks.

## System Boundaries & Ownership

### System-of-Record (SoR) Statement
Odoo serves as the system of record for scheduling, work orders, day routes, and dispatch execution. ITAD Core serves as the system of record for compliance, processing, custody, evidence, reconciliation, lots, shipments, disposition, certificates, and settlement. Routific acts solely as an optimization engine, generating route proposals without storing or owning any operational data. ITAD Core may be implemented as a standalone service or as an Odoo addon/service layer; regardless, the SoR and single-writer rules remain the same.

### Responsibilities by System
- **Odoo:** User interface for administration, backoffice operations, data entry, workflow management, and reporting; handles human edits and approvals; sole writer for scheduling/day routes within its domain.
- **ITAD Core:** Canonical service layer for business logic, data persistence, state management, and integration with external systems; enforces business rules and audit trails; sole writer for compliance/processing domains.
- **Routific:** Optimization engine for route planning and sequencing; processes input data to generate optimized proposals based on constraints and objectives.

### Write Rules
- **Single-writer principle:** Per-domain single-writer: Odoo writes scheduling/day routes; ITAD Core writes compliance/processing domains. No system may override another system's SoR data. Routific never mutates operational truth.
- **Idempotency/versioning for optimizer proposals:** Optimization requests are idempotent based on input hash; proposals are versioned to allow comparison and selection.
- **Audit/immutability expectations:** Event immutability: status transitions and operational events are append-only/immutable (event log). Controlled corrections: compliance/financial artifacts (BOL, certified weights, settlements) require void/reissue with full audit; non-critical metadata (notes, attachment references, address corrections before execution) may be updated with audit logging.

### Data Exchange Summary
- **Odoo -> Routific:** Optimization Request including stops, constraints, and objectives.
- **Routific -> Odoo:** Optimization Result with sequenced stops and ETA estimates.
- **Odoo -> ITAD Core:** Pickup_manifest bridge + BOL/receiving/processing actions + evidence pointers.
- **ITAD Core read APIs:** ITAD Core may expose read APIs for compliance truth; Odoo UI can display cached summaries but not override ITAD Core truth.
- **Routific is proposal-only:** Results are not authoritative; acceptance and execution occur in Odoo.

## B) Identifiers & Versioning

### B1) BOL# Global Uniqueness
- **Rule:** BOL numbers (bol_number) must be globally unique across all ITAD Core instances and time.
- **Rationale:** Ensures unambiguous identification of BOLs for compliance, auditing, and integration. Prevents conflicts in multi-tenant or distributed deployments.
- **Enforcement:** Database UNIQUE constraint on bol.bol_number; API rejects duplicates with 409 Conflict.

### B2) ID/Key Format Policy
- **Rule:** All primary keys use UUID v4 for internal IDs (e.g., bol.id). Human-readable identifiers (e.g., bol_number) follow format BOL-{SITE}-{YYYY}-{SEQ6}, where SITE is configurable (default "MAIN"), YYYY is current year, SEQ6 is zero-padded 6-digit sequence.
- **Rationale:** UUIDs provide collision resistance and scalability; human-readable numbers enable easy reference and sorting. Auto-generation ensures consistency without manual input errors.
- **Application:** UUIDs in all entity schemas; bol_number auto-generated if omitted in POST /api/v1/bol.

### B3) External IDs Strategy
- **Rule:** External systems (Odoo, Routific) link via external_id_map table: system (odoo|routific), entity_type (e.g., work_order, pickup_manifest), external_id (string), itad_entity_id (UUID), with UNIQUE(system, entity_type, external_id).
- **Rationale:** Enables traceability without tight coupling; supports multi-system integration while maintaining ITAD Core as SoR. Indexes optimize lookups.
- **Implementation:** Generic mapping table with constraints; APIs for CRUD operations.

### B4) Customer Requirement Profiles
- **Rule:** Requirement profiles are versioned with effective_from timestamp; snapshot (JSON) captured at BOL intake in requirement_profile_snapshot_json, with optional version/effective_from fields.
- **Rationale:** Ensures immutability for compliance; versions allow evolution of profiles without breaking historical BOLs. Snapshots prevent external changes from affecting past records.
- **Representation:** BOL schema includes requirement_profile_version (text), requirement_profile_effective_from (timestamptz), requirement_profile_snapshot_json (jsonb).

## C) Workflow (Gates, Stages, Closure)

### C1) GATE_TYPE List
- **Minimum Set:** REQUIREMENTS_CONFIRMED, PAPERWORK_VERIFIED, STAGING_ZONE_ASSIGNED, RECEIVING_ANCHOR_RECORDED, WORKSTREAMS_OPENED, BOL_CLOSED.
- **Rationale:** Defines key lifecycle events for BOL and workstreams; extensible for future phases.
- **Enforcement:** Validated in API; invalid types rejected with 422.

### C2) Append-Only Rule for bol_stage_gates
- **Rule:** bol_stage_gates are immutable; no UPDATE/DELETE allowed. Corrections use void (is_void=true, void_reason, voided_gate_id).
- **Rationale:** Ensures audit trail integrity; prevents tampering with compliance history.
- **Enforcement:** DB trigger prevents updates/deletes; API enforces append-only.

### C3) Gate Transition Matrix
- **BOL Transitions:** REQUIREMENTS_CONFIRMED → PAPERWORK_VERIFIED → STAGING_ZONE_ASSIGNED → RECEIVING_ANCHOR_RECORDED → WORKSTREAMS_OPENED → BOL_CLOSED. Invalid sequences rejected.
- **Workstream Transitions:** WORKSTREAM_OPENED → WORKSTREAM_STARTED → WORKSTREAM_COMPLETED. WORKSTREAM_VOIDED allowed at any time.
- **Rationale:** Enforces logical order; prevents invalid states.

### C4) Two-Level Closure
- **Rule:** Workstreams close first (WORKSTREAM_COMPLETED); BOL closes only after all required workstreams completed.
- **Rationale:** Ensures processing completeness before shipment/dispatch.
- **Ownership:** ITAD Core enforces; Odoo UI displays derived state.

### C5) requires_* Mechanism
- **Rule:** requires_battery_processing, requires_ewaste_processing set via confirm endpoint; locked by REQUIREMENTS_CONFIRMED gate.
- **Rationale:** Determines workstreams needed; locked to prevent changes post-confirmation.
- **Source:** Derived from requirement_profile_snapshot_json at intake.

## D) Receiving Anchor (Immutable)

### D1) Required Fields for Receiving Weight Record v3
- **Locked List:** occurred_at, bol_id, material_received_as, container_type, quantity, gross_weight, tare_weight, net_weight, weight_unit, scale_id, un_number (nullable), hazard_class (nullable), ddr_status, receiver_employee_id, receiver_name, receiver_signature_json, notes (nullable).
- **Signature Policy:** receiver_signature_json must include typed name and timestamp (format enforced by API/clients).
- **Rationale:** Ensures complete compliance data capture at receiving; all fields required except noted nullables.
- **Enforcement:** API rejects missing required fields with 422.

### D2) Immutability Rule
- **Rule:** Receiving records are immutable post-creation; corrections via void/reissue/compensating events only.
- **Rationale:** Maintains audit trail for chain of custody; prevents tampering with weight/condition data.
- **Mechanisms:** Void creates receiving_record_voids row; reissue creates new record with reissue_of_id and requires prior void; no update/delete endpoints.

### D3) Tare Policy
- **Rule:** tare_source must be one of: MEASURED_ON_SCALE, CONTAINER_INSTANCE_SNAPSHOT, CONTAINER_TYPE_PROFILE_SNAPSHOT, MANUAL_TARE_WITH_APPROVAL. Explicitly prohibits average tare without profile/snapshot.
- **Rationale:** Ensures accurate tare for net weight calculation; prevents guesswork in compliance-critical measurements.
- **Enforcement:** API validates tare_source; snapshot JSON required for snapshot-based sources.

### D4) Blind Receiving Mode
- **Rule:** When enabled, declared customer weights (declared_gross_weight, etc.) are hidden from receiving staff by default. Access requires compliance_admin role.
- **Rationale:** Prevents bias in weighing; ensures objective measurements.
- **Implementation:** When ITAD_CORE_BLIND_RECEIVING=true, GET endpoint redacts declared fields unless include_declared=true with X-Internal-Role=compliance_admin.

## E) Processing Domains & Taxonomy

### E1) Processing Sessions + Lines (Battery and E-waste)
- **Rule:** Processing is modeled as session + lines for Battery and E-waste. Acceptance/dispatch execution commits to Odoo. ITAD Core receives compliance artifacts later via pickup_manifest -> BOL -> receiving.
- **Battery Session:** started_at required; ended_at optional; headcount optional; lines required.
- **E-waste Session:** started_at and ended_at required; headcount required; lines required.
- **Correction Policy:** No update/delete for lines in Phase 0; corrections will be handled via void/reissue in later phases.

### E2) HARD RULE: taxonomy_item_id Required (No Free Text Categories)
- **Rule:** Every processing line must reference taxonomy_item_id; no free-text category fields are allowed in request schemas.
- **Rationale:** Ensures consistent classification and reporting across processing domains.
- **Enforcement:** API schemas require taxonomy_item_id; DB foreign keys enforce taxonomy linkage.

### E3) 3-Level Taxonomy + Governance Policy
- **Levels:** Group (INBOUND/OUTBOUND), Type (BATTERY/EWASTE), Variant (item-level classification).
- **Governance:** No DELETE endpoints; no in-place code renames. Changes are made by end-dating prior rows (effective_to) and inserting a new row with a new effective_from.
- **Audit:** taxonomy_change_log is append-only for CREATE/END_DATE/SUPERSEDE actions; domain_events emitted on POST create.

### E4) SB20 Flag at Taxonomy Level
- **Rule:** sb20_flag lives on taxonomy_item for reporting and compliance filtering.
- **Rationale:** Allows SB20/non-SB20 reporting without free-text variance.
- **Enforcement:** API supports sb20_flag filtering; taxonomy_item includes sb20_flag boolean.

## F) Reconciliation & Disputes

### F1) Reconciliation Model with Thresholds and Approvals
- **Rule:** Reconciliation runs are append-only snapshots per BOL with variance calculations (lbs and pct), thresholds (pct/lbs), and approval_required flag.
- **Enforcement:** reconciliation_run captures receiving vs processing totals, variance, thresholds, snapshot_json; UNIQUE per (bol_id, run_no); nonnegative checks.

### F2) Closure Blockers on Variance
- **Rule:** BOL closure is blocked when variance exceeds thresholds unless an approval event exists.
- **Enforcement:** reconciliation_approval_event is append-only; effective status derived from latest run + approval; blockers computed from authoritative runs/events.

### F3) Discrepancy / Dispute Workflow
- **Rule:** Discrepancy cases are append-only stateful records (OPEN/IN_DISPUTE/RESOLVED/VOIDED) that block closure while open.
- **Enforcement:** discrepancy_case with case_no per BOL, artifact_refs_json placeholder, resolution requires resolved_by/text; open disputes are closure blockers.

## G) Evidence & Chain of Custody

### G1) Evidence Artifacts Layer
- **Rule:** Evidence artifacts are immutable, content-addressed by sha256; storage_ref points to provider location; visibility controls (INTERNAL, CUSTOMER, COMPLIANCE_ONLY).
- **Enforcement:** evidence_artifact table with sha format checks, unique content pointer, metadata_json, retention_until optional.

### G2) artifact_links Structure
- **Rule:** Generic links connect artifacts to domain entities (BOL, RECEIVING_RECORD, BATTERY_SESSION, EWASTE_SESSION, ASSET, LOT, LPN, SHIPMENT, DISPOSITION, SETTLEMENT) with roles (e.g., SCALE_TICKET, POD).
- **Enforcement:** artifact_link table with UNIQUE(artifact_id, entity_type, entity_id, link_role) and visibility_override support.

### G3) Chain of Custody Append-Only
- **Rule:** Every move/scan emits a custody_event; corrections use compensating events (supersedes_event_id) without deleting history.
- **Enforcement:** custody_event table with append-only inserts, location codes/ids optional, metadata_json for scan references; timeline derived by occurred_at order.

## H) Inventory, Outbound, Downstream

### H1) Inventory Model (Locations, LPN, Lots)
- **Rule:** ITAD Core is SoR for inventory truth. Warehouse locations use stable codes per site; LPN/container instances carry status (WIP/READY/SHIPPED/QUARANTINE) plus convenience pointer to current_location_id, while movements are append-only custody_events. Lots are classification batches tied to taxonomy_item_id (hard rule from Phase 0.E) with optional bol_id traceability.
- **Enforcement:** Tables warehouse_location, lpn_container, inventory_lot, lot_lpn_membership with UNIQUE codes and append-only membership (removed_at). Status changes emit custody events. SoR wording lock: Acceptance/dispatch execution commits to Odoo. ITAD Core receives compliance artifacts later via pickup_manifest -> BOL -> receiving -> processing -> lots/LPN.

### H2) Outbound Shipments Domain
- **Rule:** Shipments capture carrier/appointment/seal/hazmat metadata and LPN contents. Dispatch emits custody_event SHIP per LPN; QUARANTINE LPNs cannot be loaded.
- **Enforcement:** outbound_shipment and shipment_lpn tables; outbound_repo.add_lpn_to_shipment blocks QUARANTINE; mark_shipment_status sets LPN status to SHIPPED and records custody_event. Artifacts (seals, hazmat docs) linked via artifact_link.

### H3) Downstream Qualification & Disposition Chain
- **Rule:** Downstream vendors must be allowlisted and certified (versioned, effective-dated). Dispositions require qualified vendor + final proof artifact; no free-text proofs; append-only history.
- **Enforcement:** downstream_vendor + vendor_certification (versioned, expires_at aware) with optional artifact_id proof; disposition_record links lot -> vendor -> shipment with final_proof_artifact_id. confirm_disposition enforces allowlist + active cert + proof artifact.

## I) Variant A Integrations - Pickup Manifest Bridge

### I1) Pickup Manifest Bridge + BOL Binding (1 manifest -> 1 BOL)
- **Rule:** pickup_manifest is an ITAD Core SoR record with state machine: DRAFT -> SUBMITTED -> BOUND_TO_BOL -> RECEIVED -> CLOSED (VOIDED terminal with reason). If bol.source_type == PICKUP, bol.pickup_manifest_id is required and unique.
- **Enforcement:** pickup_manifest schema carries immutable external Odoo refs, status check, manifest_no uniqueness, and UNIQUE(source_system, manifest_fingerprint) for dedupe. bol.pickup_manifest_id FK + partial unique index; pickup_manifest_state_event append-only for transitions; bind_manifest_to_bol enforces source_type=PICKUP and required status.

### I2) Canonical IDs, Idempotency, and Duplicate Detection
- **Rule:** Idempotency-Key header required for creates; manifest_no is human-readable unique; manifest_fingerprint (sha256 of completion tuple) prevents duplicate manifests under retries.
- **Enforcement:** Idempotency-Key passed through services; UNIQUE(source_system, manifest_fingerprint) on pickup_manifest; pickup_manifest_integration_attempt logs every inbound attempt with correlation_id/outcome.

### I3) Evidence / POD Handling
- **Rule:** POD evidence captured as hashes/refs in pod_evidence_json (immutable snapshot); artifacts link via artifact_link (Phase 0.G) without in-place edits.
- **Enforcement:** pod_evidence_json stored on pickup_manifest; visibility aligns to artifact rules (INTERNAL/CUSTOMER/COMPLIANCE_ONLY).

### I4) Geocode Cache + Confidence Gates
- **Rule:** geocode_cache is effective-dated and versioned per normalized address; gating thresholds: >=0.85 AUTO_ACCEPT, 0.60-0.85 NEEDS_REVIEW, <0.60 MANUAL_REQUIRED. New rows supersede old; no overwrite.
- **Enforcement:** geocode_cache table with UNIQUE(address_hash, effective_from), confidence check, is_active flag; geocode_gate applies thresholds; attach_geocode_snapshot_to_manifest copies snapshot into route_snapshot_json and sets gate.

### I5) Failure Modes & Observability
- **Rule:** All inbound attempts are logged with outcome (ACCEPTED/DUPLICATE_RETURNED/REJECTED/ERROR); correlation_id propagated; invalid transitions rejected with audit trail; void requires reason.
- **Enforcement:** pickup_manifest_integration_attempt table; pickup_manifest_state_event append-only; transition_manifest_status enforces matrix and void reason.

## Phase 1 Pre-Start Critical Check (Routing Caller)

- **Gate Question (must answer before coding Phase 1):** Who calls Routific — Odoo or ITAD Core?
- **Decision:** Routific is called from Odoo in Phase 1.
- **Rationale:**
  - Odoo is routing/dispatch SoR; proposals and acceptance must be owned there.
  - Idempotency and retry safety belong to the caller; Odoo manages retry/idempotency keys.
  - Raw Routific responses are stored/versioned with the SoR route object in Odoo.
  - ITAD Core only stores non-authoritative references (routific_job_id, input_hash) for traceability.
  - ITAD Core receives compliance outcomes later via pickup_manifest -> BOL -> receiving, not routing truth.
- **Consequences:**
  - Odoo owns: idempotency key, retry policy, proposal versioning, and acceptance/dispatch commits.
  - ITAD Core owns: compliance chain and artifacts; it does not own proposal state.

## J) Commercials & Pricing Addendum (SoR = Odoo; snapshots in ITAD Core)

### J1) Customer Pricing Profiles & External Refs (31.1)
- **Rule:** Odoo authors pricing profiles/catalogs/rate cards/tier rules; ITAD Core only stores immutable references and hashes. Effective dating tracks which Odoo version was used.
- **Enforcement:** pricing_external_ref table with ref_type (CUSTOMER_PRICING_PROFILE, SERVICE_CATALOG, RATE_CARD, TIER_RULESET), Odoo ids, ref_hash_sha256, effective_from/to, is_active. No pricing authored in ITAD Core.

### J2) Service Catalog / Charge Types (31.2)
- **Rule:** Charge types and catalogs live in Odoo; ITAD Core only keeps references/hashes and snapshots at settlement.
- **Enforcement:** pricing_payload_json in settlement_pricing_snapshot captures catalog/rate card rows used; metadata only, no authoring.

### J3) Rule Precedence (31.3)
- **Rule:** Precedence is locked/documented; ITAD Core stores precedence metadata inside pricing_payload_json snapshot.
- **Enforcement:** Snapshot payload stores precedence info from Odoo export; ITAD Core does not evaluate rules in Phase 0.

### J4) Rate Card Versioning + Settlement Snapshot (31.4)
- **Rule:** Settlement uses immutable pricing snapshots (append-only snapshot_no per settlement). Recomputes create new snapshots; no edits.
- **Enforcement:** settlement_pricing_snapshot with UNIQUE(settlement_id, snapshot_no) and snapshot_hash_sha256; append-only recomputation.

### J5) Basis-of-Charge Mapping (31.5)
- **Rule:** Snapshot stores computed bases (qty/facts) used for pricing to ensure auditability.
- **Enforcement:** basis_of_charge_json in settlement_pricing_snapshot records bases (e.g., WEIGHT_LBS); snapshot_hash covers bases.

### J6) Manual Adjustments Controls (31.6)
- **Rule:** Adjustments are append-only events with reason and approver; never mutate snapshots.
- **Enforcement:** settlement_adjustment_event requires reason_text + approver; amount signed; linked to settlement (optional snapshot pointer); audit timestamps.

### J7) Odoo ↔ ITAD Core Pricing Exchange Policy (31.7)
- **Rule:** Pricing exchange is reference-only: external ids + hashes + correlation are stored; authoring remains in Odoo. Idempotency/traceability required.
- **Enforcement:** pricing_external_ref ref_hash_sha256; settlement_pricing_snapshot snapshot_hash_sha256; exchange policy documented; No DELETE on refs/snapshots/adjustments.
