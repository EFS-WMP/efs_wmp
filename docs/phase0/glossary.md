# Phase 0 Glossary

## Glossary Lock Rules
- Terms SHALL be used consistently across UI, APIs, code, docs.
- New terms MUST be added explicitly to this glossary.
- If a definition conflicts with SoR policy, SoR policy wins.

## A) Phase 0 — Locked Terms (MVP / Canonical)

### BOL (Bill of Lading)
- **Definition:** A legal document that details the shipment of goods, including origin, destination, contents, and terms of transport. In ITAD operations, it serves as the primary manifest for outbound shipments.
- **SoR / Ownership:** ITAD Core is the system of record; authored and edited via Odoo UI.
- **Key Rules:**
  - Must be void/reissue for any corrections to maintain audit trail.
  - Links directly to Pickup Manifest and Shipment for chain of custody.

### Receiving Anchor (Receiving Weight Record / Receiving Event)
- **Definition:** The initial record of goods received at a facility, capturing weight, condition, and basic details upon arrival. Serves as the anchor point for chain of custody in processing.
- **SoR / Ownership:** ITAD Core is the system of record; authored via Odoo UI during receiving workflow.
- **Key Rules:**
  - Immutable once committed; corrections require void/reissue.
  - Triggers Lot creation and audit logging.

### Pickup Plan
- **Definition:** A high-level schedule defining what, when, and where pickups will occur, including customer details and constraints.
- **SoR / Ownership:** Odoo is the system of record (UI/workflow); ITAD Core references via external IDs and receives snapshot via pickup_manifest.
- **Key Rules:**
  - Basis for generating Pickup Occurrences; changes logged with audit trail.
  - Does not include operational execution details.

### Pickup Occurrence
- **Definition:** A specific instance of a pickup plan execution, tied to a date and assigned resources.
- **SoR / Ownership:** Odoo is the system of record; scheduled and tracked via Odoo UI.
- **Key Rules:**
  - Status transitions (scheduled, in-progress, completed) recorded in Odoo.
  - Links to Day Route for operational execution.

### Day Route
- **Definition:** The operational routing plan for a single day, including sequenced stops and assigned vehicle/driver.
- **SoR / Ownership:** Odoo is the system of record; Day Route/Stop/Statuses authored via Odoo.
- **Key Rules:**
  - Actual timestamps and status changes are immutable events in Odoo.
  - Route Proposals from Routific are advisory; acceptance finalizes in Odoo.

### Stop
- **Definition:** An individual location or task within a Day Route, such as a pickup or delivery point.
- **SoR / Ownership:** Odoo is the system of record; Day Route/Stop/Statuses authored via Odoo.
- **Key Rules:**
  - Planned/accepted/actual details stored in Odoo; edits logged.
  - Links to Pickup Manifest for execution tracking.

### Route Proposal
- **Definition:** An optimized sequence and timing suggestion generated by Routific for a Day Route.
- **SoR / Ownership:** Odoo is the system of record (generated by Routific, versioned in Odoo; ITAD Core may reference routific_job_id/input_hash non-authoritatively).
- **Key Rules:**
  - Proposals are advisory; acceptance in Odoo commits to Odoo.
  - Versioned for comparison; re-optimization creates new versions.

### Pickup Manifest
- **Definition:** A detailed list of items to be picked up at a stop, including quantities and descriptions.
- **SoR / Ownership:** ITAD Core is the system of record; authored via Odoo UI.
- **Key Rules:**
  - Links to BOL for shipment; audit trail required for changes.
  - Basis for receiving and processing workflows.

### Workstream
- **Definition:** A logical grouping of related operations or processes within ITAD, such as pickup, receiving, or disposition.
- **SoR / Ownership:** ITAD Core manages workstream assignments; configured via Odoo.
- **Key Rules:**
  - Used for workflow routing; changes logged in audit.
  - Does not affect operational truth directly.

### Lot
- **Definition:** A batch of items processed together, often from a single receiving event or pickup.
- **SoR / Ownership:** ITAD Core is the system of record; created during processing via Odoo.
- **Key Rules:**
  - Immutable once assigned; links to chain of custody.
  - Basis for LPN and Shipment creation.

### LPN (License Plate Number)
- **Definition:** A unique identifier assigned to a container or pallet for tracking during processing and shipping.
- **SoR / Ownership:** ITAD Core is the system of record; generated and managed via Odoo.
- **Key Rules:**
  - Unique within operational context; audit logged.
  - Links Lot to Shipment for traceability.

### Shipment
- **Definition:** A collection of Lots or LPNs prepared for outbound transport, linked to a BOL.
- **SoR / Ownership:** ITAD Core is the system of record; assembled via Odoo UI.
- **Key Rules:**
  - Status transitions audited; final shipment triggers settlement.
  - Compliance artifacts must be accurate.

### Disposition
- **Definition:** The final outcome decision for processed items, such as resale, recycling, or destruction.
- **SoR / Ownership:** ITAD Core is the system of record; decided via Odoo workflow.
- **Key Rules:**
  - Audited decision; affects settlement and reporting.
  - Immutable once set.

### Settlement
- **Definition:** The financial reconciliation and payment processing for ITAD services rendered.
- **SoR / Ownership:** ITAD Core is the system of record; calculated and approved via Odoo.
- **Key Rules:**
  - Based on audited operational data; void/reissue for corrections.
  - Compliance-critical; full audit trail required.

### BOL#
- **Definition:** Human-readable unique identifier for BOLs, following format BOL-{SITE}-{YYYY}-{SEQ6} (e.g., BOL-MAIN-2026-000001).
- **SoR / Ownership:** ITAD Core generates and enforces uniqueness.
- **Key Rules:** Globally unique; auto-generated if omitted; used for external references and reporting.

### UUID Policy
- **Definition:** All internal entity IDs use UUID v4 for collision resistance and scalability.
- **SoR / Ownership:** ITAD Core enforces in schemas.
- **Key Rules:** Primary keys are UUIDs; external IDs mapped via external_id_map.

### external_id_map
- **Definition:** Generic mapping table linking ITAD Core entities to external system IDs (Odoo, Routific).
- **SoR / Ownership:** ITAD Core maintains as single source of truth.
- **Key Rules:** Fields: system, entity_type, external_id, itad_entity_id; UNIQUE on (system, entity_type, external_id); enables traceability without coupling.

### Requirement Profile Versioning
- **Definition:** Versioned customer requirement profiles with effective_from timestamps for evolution.
- **SoR / Ownership:** Managed externally; snapshot captured in ITAD Core at intake.
- **Key Rules:** Version and effective_from stored with JSON snapshot in BOL for immutability.

### Requirement Profile Snapshot
- **Definition:** Immutable JSON copy of customer requirements captured at BOL creation.
- **SoR / Ownership:** ITAD Core stores snapshot.
- **Key Rules:** Prevents external changes from affecting historical BOLs; includes version metadata.

### Chain of Custody (basic)
- **Definition:** The chronological record of custody transfers for items from pickup through disposition.
- **SoR / Ownership:** ITAD Core maintains the authoritative chain; events logged immutably.
- **Key Rules:**
  - Every transfer generates an audit event.
  - Used for compliance and reporting.

### Gate
- **Definition:** An immutable event marking a lifecycle transition in BOL or workstream workflows.
- **SoR / Ownership:** ITAD Core appends gates; no external writes.
- **Key Rules:** Append-only; void for corrections.

### bol_stage_gates
- **Definition:** Append-only table storing BOL lifecycle gates with timestamps and payloads.
- **SoR / Ownership:** ITAD Core owns; used for state derivation.
- **Key Rules:** No updates/deletes; enforces transition matrix.

### Workstream
- **Definition:** A logical grouping of processing operations (e.g., BATTERY, EWASTE) required for a BOL.
- **SoR / Ownership:** ITAD Core manages; created based on requires_* flags.
- **Key Rules:** Must complete before BOL closure.

### requires_* Flags
- **Definition:** Boolean flags (requires_battery_processing, requires_ewaste_processing) indicating processing needs.
- **SoR / Ownership:** ITAD Core sets via confirm endpoint; locked by gate.
- **Key Rules:** Derived from requirement profile; immutable after confirmation.

### Closure Criteria
- **Definition:** Conditions required for BOL closure (receiving anchor, requirements confirmed, workstreams completed).
- **SoR / Ownership:** ITAD Core enforces in close API.
- **Key Rules:** All criteria must be met; API returns blockers if not.

### Tare Source
- **Definition:** Source of tare weight: MEASURED_ON_SCALE, CONTAINER_INSTANCE_SNAPSHOT, CONTAINER_TYPE_PROFILE_SNAPSHOT, MANUAL_TARE_WITH_APPROVAL.
- **SoR / Ownership:** ITAD Core validates in receiving API and DB constraints.
- **Key Rules:** Must be specified; snapshot JSON required for snapshot-based sources; "average tare" is prohibited without a profile/snapshot.

### Declared Weights
- **Definition:** Customer-provided weight estimates (gross, tare, net) for blind receiving.
- **SoR / Ownership:** ITAD Core stores; redacted in blind receiving mode by default.
- **Key Rules:** Hidden from receiving staff; available only to compliance_admin when blind receiving is enabled; used for compliance comparison.

### Blind Receiving
- **Definition:** Receiving process where declared weights are hidden to prevent bias.
- **SoR / Ownership:** ITAD Core implements redaction.
- **Key Rules:** Enabled via ITAD_CORE_BLIND_RECEIVING; declared weights require X-Internal-Role=compliance_admin with include_declared=true.

### Void/Reissue
- **Definition:** Correction mechanisms for immutable records (void marks invalid; reissue creates corrected copy).
- **SoR / Ownership:** ITAD Core appends void/reissue events.
- **Key Rules:** Maintains audit trail; no direct updates/deletes; reissue requires the original to be voided first.


### Taxonomy Group / Type / Variant
- **Definition:** Three-level classification for processing items: Group (INBOUND/OUTBOUND), Type (e.g., BATTERY, EWASTE), Variant (item-level classification).
- **SoR / Ownership:** ITAD Core governs taxonomy.
- **Key Rules:** No deletes; code changes require end-dating and new rows with new effective_from.

### Taxonomy Item
- **Definition:** Variant-level taxonomy record used by processing lines via taxonomy_item_id.
- **SoR / Ownership:** ITAD Core.
- **Key Rules:** Must be referenced by all processing lines; sb20_flag stored here.

### Processing Session
- **Definition:** A time-bounded unit of work for Battery or E-waste processing with associated lines.
- **SoR / Ownership:** ITAD Core.
- **Key Rules:** Session created with line items; no update/delete of lines in Phase 0.

### Processing Line
- **Definition:** A line item within a processing session referencing taxonomy_item_id, quantity, and weight.
- **SoR / Ownership:** ITAD Core.
- **Key Rules:** taxonomy_item_id required; no free-text categories.

### SB20 Flag
- **Definition:** Boolean attribute indicating SB20 classification for reporting at taxonomy_item level.
- **SoR / Ownership:** ITAD Core.
- **Key Rules:** Used for filtering and reporting; stored on taxonomy_item.

### Reconciliation Run
- **Definition:** Append-only snapshot comparing receiving vs processing totals with variance, thresholds, and approval_required flag.
- **SoR / Ownership:** ITAD Core.
- **Key Rules:** Unique run_no per BOL; variance_pct uses 0 when receiving_total_net_lbs=0; approvals are append-only events.

### Reconciliation Approval Event
- **Definition:** Append-only decision (APPROVE/REJECT) for a reconciliation run.
- **Key Rules:** Status derived from latest run + latest approval; over-threshold without approve blocks closure.

### Variance / Threshold
- **Definition:** Variance_lbs = receiving - processing; variance_pct = variance/receiving. Thresholds set pct/lbs for approval requirements.
- **Key Rules:** Nonnegative inputs; thresholds drive approval_required flag.

### Discrepancy Case
- **Definition:** Append-only dispute record (OPEN/IN_DISPUTE/RESOLVED/VOIDED) that blocks closure while open.
- **Key Rules:** case_no per BOL; resolution requires resolved_by + resolution_text; artifact_refs_json placeholder for Phase 0.G.

### Threshold Approval
- **Definition:** Approval decision that clears reconciliation over-threshold blocker.
- **Key Rules:** Recorded via reconciliation_approval_event; never destructive.


### Evidence Artifact
- **Definition:** Immutable evidence metadata record (sha256 content address, storage provider/ref/version, visibility, retention).
- **Key Rules:** Unique on sha256 + storage_provider + storage_ref; visibility controls access; metadata_json captures context.

### Artifact Link
- **Definition:** Generic link from artifact to domain entity with role (e.g., SCALE_TICKET, POD, CERTIFICATE_PDF).
- **Key Rules:** Unique per artifact/entity/role; optional visibility override.

### Custody Event
- **Definition:** Append-only event capturing moves/scans across entities (BOL, RECEIVING_RECORD, LOT, LPN, SHIPMENT, ASSET, CONTAINER).
- **Key Rules:** No deletes; corrections via supersedes_event_id compensating events; ordered by occurred_at.

### Visibility
- **Definition:** Access scope for artifacts (INTERNAL, CUSTOMER, COMPLIANCE_ONLY).
- **Key Rules:** Enforced in ITAD Core when listing artifacts; overrides possible per link.

### Retention
- **Definition:** Optional retention_until timestamp for artifacts to support retention policies.
- **Key Rules:** Stored on evidence_artifact; content remains immutable.

### Warehouse Location
- **Definition:** Stable code per site representing dock/scale/storage/quarantine/outbound areas.
- **SoR / Ownership:** ITAD Core.
- **Key Rules:** Unique per site; deactivate instead of delete; drives custody_event movements.

### LPN / Container Instance
- **Definition:** Unit of movement (pallet/bin/drum) identified by lpn_code with status (WIP/READY/SHIPPED/QUARANTINE) and current_location_id pointer.
- **SoR / Ownership:** ITAD Core.
- **Key Rules:** Movements emit custody_event; status changes recorded (no destructive deletes); unique lpn_code.

### Inventory Lot
- **Definition:** Classified batch tied to taxonomy_item_id (hard rule) with optional bol_id linkage and append-only LPN membership history.
- **SoR / Ownership:** ITAD Core.
- **Key Rules:** membership uses added_at/removed_at; no free-text categories; status (WIP/READY/SHIPPED/QUARANTINE).

### Outbound Shipment
- **Definition:** Outbound load with shipment_no, carrier/seal/hazmat metadata and LPN contents.
- **SoR / Ownership:** ITAD Core.
- **Key Rules:** QUARANTINE LPNs blocked from loading; dispatch writes custody_event SHIP per LPN; artifacts linked via artifact_link.

### Downstream Vendor
- **Definition:** Allowlisted downstream receiver with vendor_code and metadata.
- **SoR / Ownership:** ITAD Core.
- **Key Rules:** allowlist_flag + active status required; certifications versioned; used by disposition confirmation.

### Vendor Certification
- **Definition:** Versioned certification for downstream_vendor (R2/eStewards/DOT/etc.) with effective/expiry dates and optional proof artifact.
- **SoR / Ownership:** ITAD Core.
- **Key Rules:** No in-place rename; expire by effective_to/expires_at; artifact_id stores proof.

### Disposition Record
- **Definition:** Decision linking lot -> vendor (and optionally shipment) with disposition_type/status and final_proof_artifact_id.
- **SoR / Ownership:** ITAD Core.
- **Key Rules:** Confirmation requires vendor qualification + proof artifact; append-only history with void fields for corrections.

### Final Proof Artifact
- **Definition:** Evidence artifact proving downstream disposition (e.g., certificate of recycling/destruction).
- **SoR / Ownership:** ITAD Core.
- **Key Rules:** Stored as evidence_artifact; referenced by disposition_record.final_proof_artifact_id; visibility may be COMPLIANCE_ONLY.

### Pickup Manifest
- **Definition:** Bridge record created in ITAD Core when an Odoo pickup is completed; carries external Odoo refs, routing snapshot, POD evidence, idempotency fingerprint, and state machine.
- **SoR / Ownership:** ITAD Core (PICKUP binding SoR); external Odoo IDs are immutable references only.
- **Key Rules:** State machine DRAFT -> SUBMITTED -> BOUND_TO_BOL -> RECEIVED -> CLOSED; VOIDED requires reason; if bol.source_type == PICKUP then bol.pickup_manifest_id is required and unique.

### Manifest Fingerprint
- **Definition:** sha256 hash of canonical completion tuple (e.g., odoo_stop_id + completion timestamp + driver_id + vehicle_id) used for deduplication/idempotency.
- **SoR / Ownership:** ITAD Core.
- **Key Rules:** UNIQUE(source_system, manifest_fingerprint); retries return existing manifest; stored alongside Idempotency-Key.

### Geocode Cache
- **Definition:** Effective-dated cache of normalized address geocodes with confidence and provider metadata.
- **SoR / Ownership:** ITAD Core.
- **Key Rules:** UNIQUE(address_hash, effective_from); confidence gates: >=0.85 AUTO_ACCEPT, 0.60-0.85 NEEDS_REVIEW, <0.60 MANUAL_REQUIRED; new rows supersede old without overwrite.

### Correlation ID
- **Definition:** Request identifier propagated from Odoo inbound calls to ITAD Core for tracing/observability.
- **SoR / Ownership:** ITAD Core stores correlation_id on pickup_manifest and integration attempts.
- **Key Rules:** Included on pickup_manifest_integration_attempt and state events for debugging/retry analysis.

### Integration Attempt
- **Definition:** Logged inbound submission attempt for pickup_manifest with outcome (ACCEPTED/DUPLICATE_RETURNED/REJECTED/ERROR) and payload snapshot.
- **SoR / Ownership:** ITAD Core.
- **Key Rules:** Append-only; stores correlation_id and idempotency_key; used for failure-mode analysis and support.

### POD Evidence
- **Definition:** Proof-of-delivery artifacts (signature/photo/note) captured at pickup and referenced by pickup_manifest.
- **SoR / Ownership:** ITAD Core (hash/ref stored in pod_evidence_json; artifacts linked via evidence_artifact + artifact_link).
- **Key Rules:** Immutable snapshot; visibility follows artifact rules (INTERNAL/CUSTOMER/COMPLIANCE_ONLY); append-only.

### Pricing External Ref
- **Definition:** Immutable pointer to Odoo pricing objects (customer profile, service catalog, rate card, tier ruleset) with hash and effective dating.
- **SoR / Ownership:** Odoo authors pricing; ITAD Core stores references only.
- **Key Rules:** UNIQUE(ref_type, model, id, effective_from); ref_hash_sha256 required; no DELETE in Phase 0.

### Settlement
- **Definition:** Financial closure record for a BOL tying pricing snapshot and adjustments to final amounts.
- **SoR / Ownership:** ITAD Core.
- **Key Rules:** Status DRAFT/CALCULATED/APPROVED/ISSUED/VOIDED; append-only snapshots/adjustments; no destructive deletes.

### Settlement Pricing Snapshot
- **Definition:** Immutable pricing snapshot used for settlement, including external refs, pricing payload, bases, computed lines, and hash.
- **SoR / Ownership:** ITAD Core.
- **Key Rules:** UNIQUE(settlement_id, snapshot_no); snapshot_hash_sha256; recomputation creates snapshot_no+1 (append-only).

### Basis Of Charge
- **Definition:** Facts/quantities used to price (e.g., WEIGHT_LBS, COUNT).
- **SoR / Ownership:** Captured in settlement_pricing_snapshot.basis_of_charge_json.
- **Key Rules:** Stored in snapshot hash; immutable once recorded.

### Adjustment Event
- **Definition:** Append-only manual pricing adjustment (credit/debit/surcharge/discount) with reason and approver.
- **SoR / Ownership:** ITAD Core.
- **Key Rules:** reason_text + approver required; signed amount allowed; never mutates snapshot.

### Rate Card Version
- **Definition:** Versioned rate card reference from Odoo captured at settlement time.
- **SoR / Ownership:** Odoo authors; ITAD Core stores reference via pricing_external_ref and snapshot payload.
- **Key Rules:** effective dating in pricing_external_ref; included in settlement snapshot hash.

### Routific Proposal
- **Definition:** Optimization proposal for routes/stops generated by Routific.
- **SoR / Ownership:** Odoo (proposals are stored/versioned in Odoo; acceptance/dispatch commits to Odoo).
- **Key Rules:** ITAD Core may store routific_job_id/input_hash as non-authoritative traceability only; routing truth stays in Odoo.

### routific_job_id / input_hash
- **Definition:** Traceability fields linking to Routific calls and inputs.
- **SoR / Ownership:** Stored with the Odoo route/stop object; ITAD Core may store as references only.
- **Key Rules:** Non-authoritative in ITAD Core; used for trace/debug, not for routing state.

## B) Phase 1+ - Extended Terms (Not Locked Yet)
- **Evidence Artifact:** Digital or physical record supporting processing decisions; to be finalized in Phase 1.
- **Sanitization Job:** Process to securely erase data from devices; to be finalized in Phase 1.
- **Media:** Types of storage media handled (HDD, SSD, etc.); to be finalized in Phase 1.
- **Taxonomy Item:** Classification for items in processing; to be finalized in Phase 1.
- **Staging Zone:** Temporary holding area for items; to be finalized in Phase 1.
- **Strict “every movement MUST generate CoC event”:** Enhanced chain of custody rules; to be finalized in Phase 1.
